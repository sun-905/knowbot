# RETRIEVAL_THRESHOLD 与 TOP_K 选取说明

## 一、当前取值

| 参数 | 当前值 | 选取依据 |
|------|--------|---------|
| `RETRIEVAL_THRESHOLD` | **0.43** | 纯检索扫描，20 题标注集，安全区间中点 |
| `RETRIEVAL_TOP_K` | **5** | 兜底默认值，被 K_floor/K_opt 覆盖前使用 |
| `RETRIEVAL_TOP_K_FLOOR` | **0**（待校准） | P95(gt_rank)+1，简单查询用 |
| `RETRIEVAL_TOP_K_OPT` | **0**（待校准） | 窄窗忠实度消融最优，复杂查询用 |

> `K_FLOOR=0` 或 `K_OPT=0` 时，在线层回退到 `RETRIEVAL_TOP_K`。
> 校准后设置这两个值，系统将自动按查询类型选择 K。

## 二、三层选取方案总览

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1（离线，一次性）                                  │
│  纯检索扫描 → P95(gt_rank) + 1 = K_floor                │
│  输入：标注集（只需标"答案在哪个文档"）                    │
│  输出：统计下界，100% 确定性，不调 LLM                    │
├─────────────────────────────────────────────────────────┤
│  Layer 2（离线，定期执行）                                │
│  窄窗口 [K_floor, K_floor+4] 忠实度消融 → K_opt          │
│  Judge ≠ 被测 LLM（DeepSeek生成→Qwen评判，交叉验证）     │
│  输入：问题集（不需要标准答案，只需忠实度可评估）          │
│  输出：经验最优 K，副产品校准分层摘要阈值                  │
├─────────────────────────────────────────────────────────┤
│  Layer 3（在线，每次查询）                                │
│  查询类型感知 → 简单类用 K_floor，复杂类用 K_opt         │
│  关键词规则，确定性，可解释                               │
└─────────────────────────────────────────────────────────┘
```

---

## 三、Layer 1 详解：P95(gt_rank) + 1

### 3.1 为什么用 P95 替代 max

`max(gt_rank)` 的弱点是单个 outlier 就能绑架全局结论——50 道题中 49 道 gt_rank 在 1-3，1 道 gt_rank=18，max+2=20，所有题的上下文都被撑大了 15 条。

P95 直接问"95% 的用户体验如何"而非"最倒霉的用户有多惨"。那 5% 的 tail 由追问澄清和兜底回复处理。

### 3.2 方法

对每道有答案的题，跑检索+重排序（不调 LLM），记录 gt_rank——标注文档在重排序后的排名。取所有 gt_rank 的 P95 分位数。

```
K_floor = P95(gt_rank) + 1
```

+1 是统计余量。P95 本身已去掉 5% outlier，不需要 +2。

### 3.3 执行

```bash
cd backend
python -m tests.rag_eval.pure_retrieval_scan
```

## 四、Layer 2 详解：窄窗口忠实度消融

### 4.1 为什么只测窄窗口

Layer 1 已经给出了统计下界——K < K_floor 时正确答案大概率不在上下文里，测那些 K 值是浪费。只需在 [K_floor, K_floor+4] 内找最优。

### 4.2 为什么只用忠实度

忠实度（Faithfulness）评估的是"答案里每个陈述是否在给定上下文中找到依据"。它不需要标准答案，只需要 (问题, 上下文, 答案) 三元组。标注成本远低于需要"标准答案"的方案。

### 4.3 Judge 独立性

- DeepSeek 做主生成 → Qwen 做 Judge（主方向）
- Qwen 做生成 → DeepSeek 做 Judge（交叉验证）
- 取两次评估的平均值消除 Judge 偏见

同一模型自评存在系统性盲区——两者对同一段上下文有相同的理解偏差，Judge 看不出生成模型的幻觉。

### 4.4 决策规则

绘制 K vs Faithfulness 曲线。找到第一个 ≥ 0.95 × 最高忠实度 的 K，记为 K_opt。

若 K_opt 与 K_floor 的忠实度差异 < 0.02，取 K_opt = K_floor。

### 4.5 执行

```bash
cd backend
python -m tests.rag_eval.calibrate_top_k <K_floor>
```

### 4.6 副产品：校准分层摘要阈值

同一批 Faithfulness 数据可用于校准 context_guard.py 中的分层阈值（当前硬编码 core≥0.85 / ref≥0.70）。统计"不可验证陈述"和"可验证陈述"的 rerank_score 分布，重新确定分界线。

## 五、Layer 3 详解：查询类型感知动态 K

### 5.1 分类规则

| 查询类型 | 判断依据 | 使用 K |
|----------|---------|--------|
| 简单事实类 | 短查询、单一实体、无比较词 | K_floor |
| 复杂/多跳类 | 含比较词、分析词、长查询(>30字) | K_opt |
| 兜底 | 其他 | K_opt（安全侧） |

比较/多跳关键词：对比、比较、区别、差异、不同、哪个更好、优缺点、vs、为什么、如何解决、怎么办、流程、条件、如果…怎么办

### 5.2 为什么不用分数间隙

分数间隙法有归一化失真问题——min-max 归一化放大了间隙，而间隙本身并不对应语义鸿沟。且 `drop_threshold=0.5` 是魔法数字，没有校准路径。

查询类型法优势：确定性（同一查询永远相同 K）、可解释（"因为检测到对比类查询"）、可校准（分类规则可对照 Faithfulness 验证）。

### 5.3 代码位置

- `backend/app/services/query_type.py`：分类器和 `get_effective_top_k()`
- `backend/app/services/chat.py`：重排序步骤调用 `get_effective_top_k()`

## 六、THRESHOLD 选取说明

### 6.1 方法

THRESHOLD 控制"检不检索"——20 条候选全部低于阈值时触发兜底。它是一个**全或无闸门**，不是逐条过滤器。

逐条过滤不可行：向量和 BM25 是粗筛，低分不代表语义不相关。在重排序前按粗筛分数砍人，等同于剥夺重排序器从低分候选里捞回正确答案的能力。

### 6.2 校准数据

用 20 道标注题跑纯检索扫描：

**知识库有答案的题**（16 道）：

| C3 会员怎么升级 | 0.5425 |
| A4 如何申请退货 | 0.5176 |
| C1 退货期限一般是多久 | 0.5041 |
| A7 X3 支持哪些开锁方式 | 0.4990 |
| A2 退货需要在几天内申请 | 0.4801 |
| A10 银卡会员积分倍率 | 0.4737 |

| **B5 退款多久到账** | **0.4313** ← 最低分 |

**知识库没有答案的题**（4 道）：

| 题目 | max_raw |
|------|---------|
| **B2 可以退货到海外地址吗** | **0.4232** ← 最高分 |
| B1 产品 X5 多少钱 | 0.4039 |

| B3 CEO 是谁 | 0.1970 |

### 6.3 分析

```
                          B2(0.4232)     B5(0.4313)
无答案的题最高分 ──────────●                ●────── 有答案的题最低分
                              ↑
                         安全区间 (0.4232, 0.4313]
                              ↑
                         取中点 → 0.43
```

- threshold **必须 ≤ 0.4313**：否则 B5 被误拦
- threshold **必须 > 0.4232**：否则 B2 拦不住
- **0.43 恰好落在两者之间**

## 七、上线后重新校准

### 7.1 前提条件

- 知识库文档 30 篇以上，主题有交叉
- 标注问题 50 题以上，每题标注正确答案所在文档名
- 覆盖：直接可答题、无答案题、多文档整合题

### 7.2 步骤

```bash
# Layer 1：P95 确定 K_floor
python -m tests.rag_eval.pure_retrieval_scan
# → 得到 K_floor 和 THRESHOLD 推荐值
# → 更新 .env: RETRIEVAL_THRESHOLD, RETRIEVAL_TOP_K_FLOOR

# Layer 2：窄窗消融确定 K_opt
python -m tests.rag_eval.calibrate_top_k <K_floor>
# → 得到 K_opt
# → 更新 .env: RETRIEVAL_TOP_K_OPT
```

## 八、总结

| 参数 | 来源 | 用途 | 频率 |
|------|------|------|------|
| `RETRIEVAL_THRESHOLD` | 纯检索扫描（安全区间法） | 控制"检不检索"（全或无闸门） | 一次性 |
| `RETRIEVAL_TOP_K_FLOOR` | P95(gt_rank)+1 | 简单查询的安全下界 | 一次性 |
| `RETRIEVAL_TOP_K_OPT` | 窄窗忠实度消融（跨模型 Judge） | 复杂查询的经验最优 | 定期（换模型时） |
| 分层摘要阈值 | 忠实度数据副产品 | 噪音过滤强度 | 定期（随 K_opt） |
