# AI 智能客服系统 — 项目说明

> 基于 RAG（检索增强生成）的企业级 AI 智能客服系统

## 项目定位

将企业知识库文档转化为可对话的智能助手。用户用自然语言提问，系统从知识库中检索相关内容，由 LLM 生成有据可查的回答。

## 技术架构

| 层 | 技术 | 说明 |
|---|------|------|
| 前端 | React 19 + Ant Design 6 + Vite 8 | SPA, 暗色科幻主题 |
| 后端 | FastAPI (Python 3.10+) | 异步 Web 框架 |
| 数据库 | MySQL 8.0 | 业务数据持久化 |
| 向量库 | Qdrant | 文档向量存储与检索 |
| 大模型 | DeepSeek (主) + Qwen (备) | LLM 生成，双模型兜底 |
| 嵌入 | BGE-large-zh-v1.5 (BAAI) | 文本向量化 |
| 重排序 | BGE-Reranker-v2-m3 | Cross-Encoder 精排 |

### 技术选型原因

| 选择 | 对比方案 | 理由 |
|------|----------|------|
| **DeepSeek** | GPT-4o / Qwen-Max | ① OpenAI SDK 兼容，迁移成本低 ② 中文理解能力与 GPT-4 相当 ③ 价格 ¥1/MTok（输入），性价比远超 GPT-4o ④ 128K 上下文窗口，满足长文档场景 |
| **Qdrant** | Chroma / Milvus / FAISS | ① 原生异步 Python SDK，与 FastAPI 完美兼容 ② 支持 payload 过滤（按 kb_id 筛选），多知识库路由无需额外逻辑 ③ 单二进制部署，无外部依赖 ④ HNSW 索引 + Cosine 距离，中文语义检索精度高 |
| **BGE-large-zh-v1.5** | text2vec / m3e / OpenAI Embedding | ① MTEB 中文榜单排名第一（1024 维） ② 本地部署，无 API 调用成本 ③ Apache 2.0 开源协议 ④ GPU/CPU 灵活切换 |
| **混合检索（向量+BM25）** | 纯向量 / 纯关键词 | 向量擅长语义匹配但漏关键词精确匹配，BM25 恰好互补。RRF 融合后召回率显著高于单一方案 |
| **FastAPI** | Flask / Django | 原生异步（asyncio），SSE 流式响应零额外依赖，Pydantic 自动校验 + 自动生成 Swagger 文档 |
| **React 19 + Ant Design 6** | Vue / Svelte | 生态成熟，Ant Design 6 企业级组件开箱即用，Zustand 轻量状态管理比 Redux 少 90% 样板代码 |

## 业务思考：AI 工程问题处理

### 1. 检索为空 / 检索质量不足

**问题**：用户提问后，知识库中没有相关内容，或检索结果与问题不相关。

**处理策略**：
- **阈值拦截**：检索结果全部低于 `RETRIEVAL_THRESHOLD(0.43)` 时，不强行让 LLM 编造，直接返回兜底回复"根据现有资料，我无法确认..."
- **查询改写**：在检索前增加 LLM 查询改写步骤，将"这玩意怎么退"改写为"退货流程说明"，提高命中率
- **混合检索**：向量+BM25 互补，覆盖语义相似和关键词精确匹配两种场景
- **诊断日志**：阈值未通过时输出 top-3 分数，方便发现知识库覆盖盲区
- **分层降级**：改写超时→用原查询；检索异常→保存兜底消息，不中断对话

### 2. 上下文超长（注意力稀释）

**问题**：知识库文档切片后，检索可能返回多条内容，加上历史对话，总 token 数轻松超过 LLM 上下文窗口。即使不超，长文本中关键信息也会被 LLM 忽略（Lost in the Middle 现象）。

**处理策略**：
- **分层摘要**：按重排序分数分层——高分(≥0.85)保留全文、中分(≥0.70)截取前 200 字、低分仅保留文档名
- **类型排序**：规则 > 事实 > 流程。退货规则、售后政策等硬约束排在前面，利用"首因效应"确保 LLM 优先处理
- **Token 预算硬上限**：3000 token，超出则从低分段开始截断
- **反事实强化**：文档名始终保留，即使所有内容被截断，LLM 至少知道"某文档中有相关信息，但上下文不足以回答"

### 3. LLM 幻觉

**问题**：LLM 天然倾向"即使不知道也要编一个答案"，在客服场景中尤为危险——用户可能基于虚构的政策信息做出决策。

**处理策略（纵深防御）**：

| 层级 | 措施 | 说明 |
|------|------|------|
| **Prompt 层** | SYSTEM_PROMPT 硬约束 + ❌/✅ 反幻觉示例 | "仅根据知识库回答""不知道就说不知道"——用具体错误示例让 LLM 理解边界 |
| **输入层** | 知识在先、问题在后 | 文档内容标注【来源：文档名】，先锚定知识再提问，降低 LLM 自由发挥空间 |
| **输出层** | 三层生成后验证 | Token 重叠(≥30%) + 实体匹配 + 否定检查，对每一句回答做事实核查 |
| **架构层** | 分层摘要优先保留规则 | 规则文本（退货期限、售后条件）优先于描述性文本，减少关键数字被 LLM 误改的风险 |
| **产品层** | 每条回答标注来源 | 前端展示参考文档名和匹配度，用户可自行判断可信度 |

### 4. LLM 服务不稳定

**问题**：API 可能超时、限流、或返回异常。单次失败不应导致整个对话中断。

**处理策略**：DeepSeek(主) → Qwen(备) → 静态兜底。tenacity 指数退避重试 3 次（1s→2s→4s），连续 5 次失败触发 60s 熔断避免雪崩。

## 使用 AI 编程工具的体会

本项目全程使用 **Claude Code**（Anthropic 推出的 AI 编程助手）辅助开发。

### 效率提升

- **文档对齐**：项目有 10+ 篇 Markdown 文档，代码改动后手动同步文档极易遗漏。AI 能并行比对所有文档与代码，一次性找出不一致的配置值、过时的流程描述、缺失的步骤。
- **全栈排查**：一个"前端不显示历史消息"的问题，AI 同时检查后端 SSE 事务提交逻辑、前端 Zustand 状态管理、React 18 StrictMode 副作用——跨三层定位到 `finally` 中无条件 `rollback()` 的 bug，人工排查可能需要数小时。
- **依赖地狱**：PyTorch CPU/GPU 版本问题、cryptography 版本兼容、bcrypt 接口变更——AI 对常见库的 breaking change 有记忆，能快速认出"这是 passlib 与 bcrypt 4.1+ 不兼容的已知问题"。

### 局限与体会

- **AI 擅长"找"不擅长"判"**：AI 能精确找到 bug 位置，但修复方案的取舍需要人来判断。比如 StrictMode 双重挂载问题，AI 提出 cleanup 方案，但实际测试发现 cleanup 反而造成死循环，最终还是人决定"去掉 ref guard，仅依赖 sessionId"。
- **幻觉同样存在**：AI 有时会"看到"代码中不存在的逻辑，需要人对每个结论做验证。对策是每次让 AI 改完代码后立即跑测试 + 看实际运行日志。
- **上下文是瓶颈**：大型代码仓库（本项目 ~200 个文件）超出 AI 单次上下文窗口。需要借助 CLAUDE.md 做"路书"——提前告诉 AI 项目架构、关键文件位置、编码规范，减少无效探索。

### 结论

AI 编程工具显著缩短了排查-修复-验证的循环时间，尤其在跨层问题和文档同步场景。但**最终决策权和代码质量责任仍在开发者**——AI 是副驾驶，不是自动驾驶。

- **RAG 智能问答**：混合检索（向量+BM25）+ 重排序 + 上下文守护 + LLM 流式生成
- **意图识别**：三级级联（L1规则→L2向量→L3大模型），置信度 <0.7 追问澄清
- **知识库管理**：上传文档自动解析、切片、向量化，支持多知识库
- **会话管理**：独立 Session，多轮对话上下文，历史记录查询
- **反馈系统**：点赞/踩 + 可选文字反馈
- **管理后台**：问答统计、反馈统计、意图分布、折线图
- **反幻觉机制**：分层摘要、类型排序、生成后三层验证、SYSTEM_PROMPT 硬约束

## 快速开始

详见 `运行指南.md`。

```bash
# 1. 启动基础设施
docker compose up -d mysql qdrant

# 2. 配置 API Key
cp .env.example .env
# 编辑 .env，填入 DEEPSEEK_API_KEY

# 3. 启动后端
cd backend && pip install -r requirements.txt
uvicorn app.main:app --port 8000

# 4. 启动前端
cd frontend && npm install && npm run dev
```
