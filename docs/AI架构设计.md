# AI 架构设计

## RAG 流程

```
用户消息
  │
  ▼
[1] 意图识别 (L1规则 → L2向量 → L3大模型)
  │  置信度<0.7 → 追问澄清
  │  意图=闲聊 → 自由对话模式
  │
  ▼
[2] 查询改写 (LLM 口语→专业术语)
  │  闲聊跳过 | 过短≤2字跳过 | 过长>500字跳过
  │  5s超时降级 | 异常降级 → 使用原始查询
  │  SSE: event=rewritten_query
  │
  ▼
[3] 混合检索 (向量 K=20 ‖ BM25 K=20 → RRF 融合 → 前20)
  │  全低于阈值(0.43) → 兜底回复
  │
  ▼
[4] 查询类型分类 (query_type.py)
  │  简单事实类(短查询/单一实体) → K_floor
  │  复杂类(比较/多跳/长查询) → K_opt
  │  未校准时回退到 retrieval_top_k (5)
  │
  ▼
[5] Cross-Encoder 重排序 (bge-reranker-v2-m3, 8s超时降级, GPU 优先 CPU 兜底)
  │  分数 min-max 归一化 [0,1]
  │  取前 effective_K 条
  │
  ▼
[6] 上下文守护
  │  分层摘要: core(≥0.85 全文) / ref(≥0.70 前200字) / edge(<0.70 仅文档名)
  │  类型排序: 规则 > 事实 > 流程
  │  token预算: 3000
  │
  ▼
[7] 构建 Prompt
  │  system: 分层规则 + 反幻觉示例
  │  history: 最近5轮对话
  │  user: 知识库内容（标注来源）+ 用户问题
  │
  ▼
[8] LLM 生成 (DeepSeek → Qwen兜底 → 静态兜底)
  │  tenacity 重试3次 (1s→2s→4s)
  │  熔断: 连续5次失败 → 冷却60s
  │  SSE 流式输出
  │
  ▼
[9] 生成后验证
  │  三层: token重叠(≥30%) + 实体匹配 + 否定检查
  │  全部句子
  │
  ▼
[10] 追问建议 (异步, 3s超时降级)
```

## 检索架构

```
向量检索 (Qdrant, 1024维 Cosine)     BM25 关键词检索 (内存索引)
        │                                    │
        └──── RRF 融合: 1/(60+rank) ─────────┘
                      取前20
                       │
                Cross-Encoder 重排序
                bge-reranker-v2-m3
                分数归一化 [0,1]
                       │
            取前 K 条（动态选择：简单查询=K_floor，复杂查询=K_opt，兜底=5）
```

## 嵌入模型

- **BGE-large-zh-v1.5** (BAAI)
- 输出维度: 1024
- 设备: CPU（可配置 CUDA）

## 文档入库

```
上传(PDF/MD/TXT) → create_document (status=processing, 秒返)
                 → BackgroundTasks.process_document
                    ├─ LlamaIndex SimpleDirectoryReader 解析
                    ├─ SentenceSplitter 切片 (512字, 50重叠)
                    ├─ BGE 向量化
                    ├─ Qdrant upsert
                    ├─ BM25 增量更新
                    ├─ status → ready
                    └─ 失败 → status=failed + error_msg
```

## 关键设计决策

### 1. 上传与处理分离

文档上传立即返回 201 (status=processing)，后台异步处理。启动时扫描残留的 processing 文档自动补处理。避免长连接超时，用户可实时看到状态变化。

### 2. 分层摘要 + 类型排序

防止 LLM 注意力稀释的三层压缩：高相关保留全文、中相关截 200 字、低相关仅文档名。然后按"规则 > 事实 > 流程"排序，利用首因效应确保关键约束在注意力最强处被处理。

### 3. 双模型兜底 + 熔断

DeepSeek(主) → Qwen(备) → 静态兜底消息。连续 5 次失败触发 60s 熔断，避免故障连锁。tenacity 指数退避重试处理瞬时抖动。

### 4. 查询改写

在意图识别与混合检索之间插入 LLM 查询改写步骤，将用户口语化表达（"这玩意怎么退"）转化为专业检索查询（"退货流程说明"）。改写后查询用于向量/BM25 检索和重排序，原始消息保留用于 LLM 生成和前端展示。

**降级策略**：闲聊跳过、短消息(≤2字)跳过、长消息(>500字)跳过、5 秒超时降级、LLM 异常降级。改写失败不影响主流程。

### 5. 阈值校准

通过 20 题标注集纯检索扫描，确定 RETRIEVAL_THRESHOLD=0.43（有答案最低分与无答案最高分之间的安全区间）。详见 `RETRIEVAL_THRESHOLD与TOP_K选取说明.md`。

## Prompt 设计

### 业务对话 SYSTEM_PROMPT

```
你是一个基于知识库的 AI 客服助手。你的所有知识来自系统提供的知识库文档，
你不代表任何具体公司，也不知道任何公司的内部信息。请严格遵循以下规则：

【信息边界 — 最高优先级】
1. 仅根据提供的「知识库内容」回答问题。知识库中没有的信息，你必须明确说
   "根据现有资料，我无法确认..."，绝对不要猜测或编造。
2. 如果知识库内容之间存在矛盾，请指出矛盾之处，不要擅自选择其一。
3. 禁止使用你的训练数据或常识来补充知识库中不存在的信息。
4. 关于公司内部信息（CEO是谁、员工姓名、组织架构、公司地址等），除非知识库
   中明确记载，否则你必须回答"根据现有资料，我无法确认"。

【回答格式】
5. 回答结构：先给出直接结论，再展开说明细节。
6. 引用来源时用【来源：文档名】标注。

【安全约束】
7. 涉及法律、财务、医疗建议时，必须在回答末尾加"建议您进一步咨询相关专业人士"。
8. 用户提出投诉时，先表达理解和歉意，再给出解决方案。

【输出控制】
9. 回答简洁，控制在 300 字以内（除非用户明确要求详细）。
10. 使用 Markdown 优化可读性。

【反幻觉示例】
❌ 知识库只有"产品A售价100元" → 问"产品B多少钱" → 答"产品B售价200元"（编造）
✅ 知识库只有"产品A售价100元" → 问"产品B多少钱" →
   答"根据现有资料，我无法确认产品B的价格"
❌ 知识库说"退货期限7天" → 答"退货期限一般是7-14天"（用常识补充）
✅ 知识库说"退货期限7天" → 答"根据现有资料，退货期限为7天"
```

### 闲聊对话 CASUAL_SYSTEM_PROMPT

```
你是一个基于知识库的 AI 客服助手。用户正在与你闲聊。

重要提醒：你不知道公司的内部信息（CEO、员工、地址等），
除非对方明确询问的是知识库中已有的产品/服务问题。
闲聊时不要编造任何关于"公司"的具体信息。

你可以：
1. 自然地介绍自己：你是 AI 客服助手，可以帮用户解答产品问题、查询订单、处理售后等。
2. 友好地回应问候和闲聊话题。
3. 如果对方问到你不知道的信息，诚实地说"这个我不太清楚，建议您咨询人工客服"。

保持简短、亲切、有人情味的回复。
```

### Prompt 拼接策略

```
┌─ System Prompt (按意图选择业务/闲聊) ─┐
├─ 对话历史 (最近5轮，每条 role+content)┤
├─ [闲聊] 当前日期 system 消息 ──────────┤  ← 紧邻用户问题
├─ 知识库内容 (标注来源) ────────────────┤    先定边界
├─ 用户原始问题 ────────────────────────┘    再给问题
```

**设计原则**：

1. **结构 > 叙述**：`【区块】` + 编号，LLM 逐条遵守，比自然语言描述更有效
2. **示范 > 描述**：❌/✅ 对比给出正确和错误示例——LLM 对具体案例的理解远超抽象规则
3. **模板 > 指引**：提供具体话术模板（"根据现有资料，我无法确认..."），不靠 LLM 自己措辞
4. **锚定 > 开放**：知识在先、问题在后——先建立信息边界，再提问，降低幻觉空间
5. **隔离 > 混合**：闲聊/业务用不同 system prompt——闲聊不需要反幻觉约束，业务不需要自我介绍的废话
6. **日期覆盖**：闲聊场景下日期作为独立 system 消息放在历史之后、用户问题之前，并显式告知"即使历史记录中有不同日期也要以此为准"
