# Prompt 优化与反幻觉设计

> 本文档说明 RAG 客服系统中 Prompt 优化的核心思路，以及如何通过 Prompt 设计减少 LLM 幻觉。

---

## 一、优化总览

系统中有 **5 个独立的 Prompt 场景**，每个都针对特定任务做了定向优化：

| 场景 | Prompt | 位置 | 核心策略 |
|------|--------|------|----------|
| 主对话 | `SYSTEM_PROMPT` | `chat.py:18` | 分层规则 + 反幻觉示例 |
| 闲聊 | `CASUAL_SYSTEM_PROMPT` | `chat.py:43` | 角色限定 + 日期注入 |
| 意图分类 | `_l3_llm_classify` | `intent.py:91` | 零温 + 结构化输出 + 低上限 |
| 追问生成 | `FOLLOWUP_SYSTEM_PROMPT` | `chat.py:72` | 知识库锚定 + 格式约束 |
| 澄清追问 | `_get_clarify_question` | `chat.py:339` | 规则兜底（不调 LLM） |

## 二、主对话 Prompt 优化（核心）

### 2.1 优化思路：分层规则 + 优先级标记

```
【信息边界 — 最高优先级】    ← 阻止幻觉的第一道墙
【回答格式】                ← 确保输出可追溯
【安全约束】                ← 合规性要求
【输出控制】                ← 用户体验
【反幻觉示例】              ← 用示范强化前文规则
```

**设计原理**：LLM 对 prompt 中**开头部分**的注意力最高（首因效应）。将"信息边界"放在最前面并用"最高优先级"标注，确保 LLM 在生成第一个 token 之前就已经内化了"不能编造"的约束。

**对比普通做法**：

```
❌ 常见写法：
"你是一个客服助手，请友好地回答用户问题。注意：不要编造信息，仅根据提供的资料回答。"

✅ 本项目写法：
【信息边界 — 最高优先级】
1. 仅根据提供的「知识库内容」回答问题。
   知识库中没有的信息，你必须明确说"根据现有资料，我无法确认..."，绝对不要猜测或编造。
2. 如果知识库内容之间存在矛盾，请指出矛盾之处，不要擅自选择其一。
3. 禁止使用你的训练数据或常识来补充知识库中不存在的信息。
   即使你认为某件事"应该是这样"，只要知识库里没有，就不能说。
```

**关键差异**：
- "不要编造信息" vs "你必须明确说'根据现有资料，我无法确认...'" —— 后者给出了**具体的输出模板**，LLM 不需要自己措辞
- 规则 3 直接点出 LLM 最常犯的错误——用训练数据补全 —— 并明确禁止
- 用 `【】` 和编号创造结构感，让 LLM 将规则视为 checklist

### 2.2 优化思路：反幻觉示例（Few-shot 示范）

```text
【反幻觉示例】
❌ 知识库只有"产品A售价100元" → 问"产品B多少钱" → 答"产品B售价200元"（编造）
✅ 知识库只有"产品A售价100元" → 问"产品B多少钱" → 答"根据现有资料，我无法确认产品B的价格"
❌ 知识库说"退货期限7天" → 答"退货期限一般是7-14天"（用常识补充）
✅ 知识库说"退货期限7天" → 答"根据现有资料，退货期限为7天"
```

**设计原理**：这条规则专门对抗 LLM 最隐蔽的一种幻觉——**常识性外推**。LLM 在知道"退货期限是 7 天"的情况下，可能基于训练数据中的常见模式，自动补充"一般在 7-14 天之间"。这种幻觉不会被 token 重叠检测捕获（"7 天"确实在来源中），但对用户产生了误导。

**为什么不用长示例**：每条例 2 行、共 4 条，token 开销约 120。如果写成段落叙述，LLM 反而可能学习"解释性回答"的模式而非"边界约束"。

### 2.3 优化思路：知识上下文的结构化组装

```python
# 最终发给 LLM 的 user message 结构
knowledge_text = "\n\n---\n".join([
    f"【来源：{d.get('doc_name', '未知')}】{d.get('text', '')}"
    for d in valid_docs
])

messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    *history,   # 最近 5 轮对话
    {"role": "user", "content": f"知识库内容：\n{knowledge_text}\n\n用户问题：{user_message}"},
]
```

**设计原理**：

1. **知识库内容在前，用户问题在后**：LLM 先"读资料"再"听问题"，模拟人类查阅文档后作答的认知顺序。如果反过来（先问题后知识），LLM 可能先基于自身知识形成初步答案，再选择性关注知识库内容（确认偏误）。

2. **来源标注前缀 `【来源：文档名】`**：每个 chunk 前强制标注出处，LLM 在生成时自然倾向于引用来源名，实现了"引用即验证"——如果 LLM 无法标注来源，说明这条信息可能是编造的。

3. **分离符 `---`**：用视觉分隔符区分不同来源的 chunk 边界，降低 LLM 跨 chunk 混淆不同文档信息的概率。

4. **对话历史夹在 system 和 user 之间**：system prompt 保证全对话有效，历史消息提供上下文连贯性，当前知识 + 问题提供最新锚定。

## 三、闲聊 Prompt 的场景隔离

```python
CASUAL_SYSTEM_PROMPT = """你是一个专业、友好的企业客服助手。用户正在与你闲聊。
你可以：
1. 自然地介绍自己：你是这家公司的 AI 客服助手，可以帮用户解答产品问题、查询订单、处理售后等。
2. 友好地回应问候和闲聊话题。
保持简短、亲切、有人情味的回复。"""
```

**设计原理**：闲聊和知识问答使用**完全不同的 system prompt**，防止闲聊的开放语境渗透到知识问答的约束逻辑中。

**日期注入**：闲聊时额外注入当前日期为 system 消息，紧邻用户问题：
```python
messages.append({"role": "system", "content": f"现在是{today_str}。回答任何日期时间问题都必须以此为准，即使历史记录中有不同的日期也要以此覆盖。"})
```

这条规则很重要——LLM 的训练数据有时滞性，不知道"今天"是什么日期。用 `system` 角色而非 `user` 角色注入日期信息，赋予其更高的指令优先级。

## 四、意图分类 Prompt 的防噪设计

```python
system_prompt = (
    "你是一个意图分类器。将用户消息归入以下类别之一："
    "产品咨询、售后问题、订单查询、账号问题、投诉、闲聊、其他。"
    "只返回 JSON：{\"intent\": \"...\", \"confidence\": 0.0-1.0}"
)
# ...
response = await client.chat.completions.create(
    temperature=0,     # 零温：消除随机性，分类结果可复现
    max_tokens=100,    # 低上限：阻止 LLM 输出多余的解释
)
```

**设计原理**：

- **temperature=0**：意图分类不需要创造性，零温确保相同输入 → 相同输出，结果可审计
- **max_tokens=100**：JSON 格式 `{"intent": "产品咨询", "confidence": 0.95}` 约 50 tokens。设 100 的上限既留有余量，又阻止 LLM 输出"解释为什么这样分类"的额外文本
- **只返回 JSON**：明确禁止自然语言解释，用正则 `r"\{[^}]+\}"` 提取 JSON，容忍 LLM 偶尔多输出标点

**三级降级**：规则（L1）→ 向量（L2）→ 大模型（L3）。前两层不需要调 LLM，减少了对 LLM 的依赖，也节省了 token 和延迟。

## 五、追问生成 Prompt 的知识锚定

```python
FOLLOWUP_SYSTEM_PROMPT = """
规则：
1. 追问建议必须能从知识库中找到答案（不要提知识库覆盖不到的问题）
2. 追问建议要与当前话题紧密相关，自然的下一步问题
3. 每条追问建议不超过 20 个字
4. 优先提供"怎么做"、"需要什么"、"多长时间"等有明确答案的问题
5. 回复格式：每行一个追问建议，只输出 3 行，不要编号，不要任何其他内容
"""
```

**设计原理**：约束级联，从高到低：
- 规则 1 → **能否回答**（锚定知识库，阻止生成无答案的追问诱饵）
- 规则 2 → **是否相关**（话题一致性）
- 规则 3 → **长度控制**（UI 展示友好）
- 规则 4 → **内容偏好**（倾向有明确答案的方向，而非开放性问题）
- 规则 5 → **格式约束**（可直接按行 split 解析，不依赖 JSON 解析）

**容错设计**：生成失败 → 基础追问 → 硬编码降级（`["还有其他问题吗？", "需要转接人工客服吗？"]`），确保追问建议永远不为空。

## 六、澄清追问 Prompt 的规则化

```python
def _get_clarify_question(user_msg: str, intent_result: dict) -> str:
    clarify_map = {
        "投诉": "您能具体描述一下遇到的问题吗？我会尽快帮您处理。",
        "售后问题": "请问您是想咨询退货、退款还是换货呢？",
        "产品咨询": "您是对哪款产品感兴趣呢？我可以为您详细介绍。",
        "订单查询": "请提供您的订单编号，我帮您查询。",
    }
    return clarify_map.get(intent, "抱歉，我没有完全理解您的意思，能再说得具体一些吗？")
```

**设计原理**：**完全不用 LLM 生成澄清问题**。这是一个纯规则映射函数。原因：
- 澄清追问需要**精确引导**用户提供特定信息（如订单号）
- LLM 生成的澄清问题可能过于宽泛（"请提供更多信息"），无法有效缩小信息缺口
- 确定性输出，零幻觉风险

## 七、上下文中的标签化知识注入

发给 LLM 的知识内容经过**来源标注**：

```
【来源：退货政策.pdf】退货需在收到商品后 7 天内申请，商品需保持完好无损。

---

【来源：售后流程.md】申请退货步骤：1. 登录账户 2. 进入"我的订单" 3. 点击"申请退货"。
```

**与普通做法的对比**：

```
❌ 普通拼接：
"退货需在收到商品后 7 天内申请... 申请退货步骤：1. 登录账户..."

问题：LLM 不知道这两段来自不同文档，可能混淆或合并不一致的规则。
```

```
✅ 标签化：
每条知识前有【来源：文档名】，之间有 --- 分隔。

效果：
1. LLM 能区分不同来源，输出时自然引用文档名
2. 用户看到【来源：退货政策.pdf】知道信息可追溯
3. 存在矛盾时（如两个文档给出不同期限），LLM 更容易识别并指出
```

## 九、Prompt 优化的核心原则总结

1. **结构 > 叙述**：用 `【区块】` 和编号而非段落叙述，LLM 更容易逐条遵守
2. **示范 > 描述**：用 ❌/✅ 对比替代"你应该怎么做"的叙述
3. **模板 > 指引**：给出"根据现有资料，我无法确认..."的具体话术模板，而非"不知道就说不知道"
4. **规则 > 判断**：硬编码澄清追问而非让 LLM 生成，消除不确定性
5. **锚定 > 开放**：知识库内容放在用户问题之前，先锚定知识边界再回答
6. **隔离 > 混合**：闲聊和业务使用完全不同的 system prompt，防止语境污染
